# ADR-0001: Python 3.12 as the implementation language

- **Status:** Accepted
- **Date:** 2026-05-23
- **Decider(s):** Karthik Gajjala

## Context

The renderer service, fixture loaders, Netbox client wrapper, and QEMU launcher all need an implementation language. The criteria: mature ecosystem (HTTP frameworks, schema validation, RDMA/network tooling, Netbox client libraries), strong static typing support, broad hiring base, durable enough to outlive a generation.

Candidates considered: Python, Go, Rust.

## Decision

**Python 3.12.**

Python's ecosystem dominates infrastructure tooling — pynetbox, Ansible, Pydantic, FastAPI, structlog, OpenTelemetry, the entire RDMA/RDMA-core tool family. Static typing via type hints + `mypy --strict` (ADR-0004) addresses Python's historical weakness around runtime type errors. 3.12 is mature, ecosystem-supported, and the long-term-stable target for libraries we depend on.

3.13's free-threaded mode is interesting but ecosystem support is partial as of mid-2026; we pin to 3.12 for ecosystem maturity.

## Consequences

**Easier:**
- Hiring (Python is the most-known language in infrastructure).
- Library availability for every concern we have.
- Rapid prototyping.

**Harder:**
- Runtime performance vs Go/Rust — acceptable for a config renderer (not on the data path of any host's traffic).
- GIL constraints — acceptable; our concurrency story is "FastAPI async + asyncio.to_thread for blocking I/O," not "thousands of threads."

**Risks introduced:**
- Type-system gaps: Python's type system is strong but escapable. Mitigation: `mypy --strict` enforced by CI (ADR-0004).
- Dependency drift: large surface. Mitigation: pinned via `uv.lock` (ADR-0002), Dependabot.

**Triggers for re-evaluation:**
- If we find ourselves writing hot-path code (we shouldn't; the renderer is not on the data plane), revisit.
- If the GIL becomes a meaningful bottleneck under realistic load, revisit (3.13 free-threading would be the answer, not a language change).
